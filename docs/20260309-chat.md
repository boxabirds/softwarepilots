## VibeSDK Assessment

Researched [Cloudflare VibeSDK](https://github.com/cloudflare/vibesdk) against education platform needs.

**What it is:** Open-source text-to-app platform. Users describe what they want in natural language, an AI agent generates a full React/TypeScript/Tailwind app, runs it in a sandboxed Cloudflare Container for live preview, and can deploy to Workers for Platforms. MIT licensed, 4.8k stars, v1.5.0 (Feb 2026).

**Architecture highlights relevant to the platform:**
- Durable Objects pattern matches the per-learner agent architecture
- D1 with Drizzle ORM for persistence
- AI Gateway for rate limiting and cost control across multiple LLM providers
- Sandboxed containers for isolated execution
- Programmatic TypeScript API (`@cf-vibesdk/sdk`) via `PhasicClient.build()` — could let learners submit specs and get evaluated apps back
- Phase-wise generation model (planning, foundation, core, styling, integration, optimisation)

**Fit assessment:**
- **Not a fit for Modules 1–3.** VibeSDK generates complete apps from natural language — the opposite of the compiler moment's need for minimal, controlled, instruction-level execution.
- **Strong fit for Modules 6–7 and the final project.** Fork it, strip the consumer UI, wrap with evaluator rubric system and learner progress tracking.

**Constraints:** Requires Cloudflare Workers Paid Plan, Workers for Platforms subscription, Advanced Certificate Manager for preview subdomain wildcards. Defaults to Google Gemini. Container instances range 256MB to 12GB/4vCPU — cost per concurrent learner needs modelling.

## Browser-Based Code Execution: Three Tiers

### Tier 1: Sandpack (CodeSandbox) — Modules 1–3

Best fit for the compiler moment and early exercises.

- Open source, MIT licensed, self-hostable bundler
- Total control over visible/editable/read-only files
- Live console output piped from sandboxed iframe
- Used by React docs, Josh Comeau's courses, Svelte/Angular/Nuxt tutorials
- Pure client-side bundling — zero marginal compute cost per learner
- React component API: `<Sandpack template="vanilla" files={{...}} options={{showConsole: true}} />`
- Limitation: JavaScript/TypeScript bundler, not a full OS (fine for the compiler moment)
- Now has Node.js support via Nodebox

### Tier 2: WebContainers (StackBlitz) — Modules 4–5 transition

- WebAssembly-based micro-OS running full Node.js in the browser
- Real terminal, real file system, real `npm install`
- Powers StackBlitz and Bolt.new
- **Commercial license required** for for-profit production usage — sales conversation needed
- Browser support: full in Chromium only; Firefox and Safari in beta; embedding only Chromium
- Battle-tested for programming tutorials and onboarding platforms

### Tier 3: VibeSDK / Cloudflare Containers — Modules 6–7, final project

Full Docker containers, real deployment, real app generation. Evaluates learner outputs rather than process.

### Also noted: Pyodide

Full CPython via WebAssembly in browser (Mozilla Public License). Option if the compiler moment is ever taught in Python instead of JavaScript.

## Curriculum Architecture Decisions

### Three-layer structure

Emerged from the question: "what does this course teach that a conventional software engineering course doesn't?"

- **Layer 1 — Pilotry-Native** (the product): probabilistic systems, specification as a discipline, autonomy spectrum, verification of probabilistic outputs, agentic UX, simulation, model selection, memory management, sustainable practice.
- **Layer 2 — Applied Foundations** (taught through pilotry lens): request lifecycle, DevTools, database fundamentals, security, observability.
- **Layer 3 — Reference Labs** (living documentation, not core): specific agent tools, Docker, email, canvas, WebSockets/SSE, DNS deep dives.

### Stability spectrum for evergreen design

- **Stable 5+ years:** probabilistic nature of LLMs, HTTP, database fundamentals, security principles, DNS, observability, the need for specifications, UX research concepts, browser rendering.
- **Directionally clear, specifics moving:** agent autonomy levels, MCP and tool protocols, multi-agent coordination, edge model capabilities, context windows. Teach concepts not tools.
- **Genuinely fluid (not in evergreen course):** specific coding agent UIs, specific model names, specific prompt templates, pricing.

### Two pedagogical anchors

1. **The compiler moment** (Module 2): software is precise and literal.
2. **The stochastic moment** (Module 3): AI agents are neither. Same spec, five runs, five different outputs.

### Where the pilot's value concentrates

1. **Specification** — problem definition, testable acceptance criteria, edge cases, decomposition, constraints.
2. **Judgment at the boundaries** — risk assessment, autonomy-appropriate oversight, knowing your limits.
3. **Verification and acceptance** — independent verification, the pre-launch checklist, go/no-go decisions.

## The Autonomy Spectrum

Taxonomy discussed and incorporated into Module 5:

| Level | Example | Primary failure mode |
|-------|---------|---------------------|
| Structured workflow | Ceetrix, Magenta-style, CI/CD | Gets stuck, fails silently, follows bad instructions literally |
| Tool-using agent | Claude with MCP, Cursor, Windsurf | Hallucinated tool calls, wrong tool selection, context drift |
| Semi-autonomous agent | Claude Code autonomous, Devin | Confident wrong architecture, scope creep, tech debt |
| Multi-agent swarm | Agent teams (coder + reviewer + tester) | Groupthink hallucination, circular validation, architectural incoherence |
| Self-modifying system | OpenClaw/NanoClaw, Gas Town-style | Goal drift, unintended optimisation, loss of human oversight |

Pilot's role shifts at each level: operator → supervisor → director → governor.

## Revised 3x3 Proposition

Iterated twice. Final version (BBC-length single sentences):

**CONTEXT**
1. AI agents now generate working software from plain English.
2. Hundreds of millions of domain experts can finally build solutions to problems only they understand.
3. The tool doing the building is probabilistic — it is confident, inconsistent, and frequently wrong.

**PROBLEM**
4. Non-technical creators are shipping software they cannot evaluate, secure, or maintain.
5. Professional engineers have vital skills but no defined role in a world where machines write the code.
6. When AI-generated software causes harm, nobody is clearly accountable.

**SOLUTION**
7. Engineers must evolve into software pilots — skilled practitioners who oversee, intervene, and take responsibility.
8. Employers must hire for judgment, specification, and verification — not just coding ability.
9. Education must teach pilotry as a discipline: the machine beneath, the probabilistic machine above, and the human accountable for both.

## Wellbeing by Design

Emerged from personal experience with the Ceetrix build. Key insight: AI agents remove the natural friction that previously limited overwork. The agent is always available, every idle thought becomes an actionable prompt, and it feels productive rather than compulsive.

Threaded through the curriculum at five touchpoints:
- **Design Principles** — "Wellbeing by design" as structural requirement
- **Module 1** — "The pilot's most underrated skill: stopping"
- **Module 4** — Specification quality degrades with fatigue
- **Module 6** — "The one more thing trap"
- **Module 8** — Pilot fitness on the pre-launch checklist
- **Module 9** — Full section on sustainable pilotry with FAA Part 117 precedent

The aviation analogy: duty limits exist not because pilots want to stop flying, but because fatigued pilots kill people. The regulation exists precisely because the individual cannot be trusted to self-regulate when the work feels important and the capability is available.
